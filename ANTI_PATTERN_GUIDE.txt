ERRORE #1: Schema Prisma non sincronizzato con DB
---
PROBLEMA:
  - Aggiungi colonna a schema.prisma
  - Container parte, ma DB non ha la nuova colonna
  - Risultato: "column `reserved` does not exist" (500 errors)

ROOT CAUSE:
  - docker-compose.yml montava schema Prisma in read-only: ./server/prisma:ro
  - Questo blocca le migrazioni Prisma

SOLUZIONE:
  ✅ RIMUOVI `./server/prisma:ro` da docker-compose.yml
  ✅ Lascia Prisma gestire migrazioni liberamente
  ✅ Priorizza schema.prisma come FONTE DI VERITÀ

CODICE CORRETTO (docker-compose.yml):
  volumes:
    - ./data/molino:/data/molino          # Database + backups
    - ./public:/app/public               # HTML/CSS/JS (no schema!)
    # ❌ NON MONTARE: ./server/prisma:ro
    # ❌ NON MONTARE: ./server/src:ro

PREVENZIONE:
  - Prima di ogni deploy, verifica schema.prisma
  - Esegui: npx prisma migrate dev (localmente)
  - Poi: docker compose build
  - Verifica: docker compose logs | grep -i "migrat"

ERRORE #2: Startup script bloccato su Prisma
---
PROBLEMA:
  - Container parte ma rimane stuck su "Running Prisma migrations..."
  - Timeout 3+ minuti, client app non carica

ROOT CAUSE:
  - docker-start.sh eseguiva: npx prisma migrate deploy
  - Con --skip-generate, ma blocca comunque se migrazioni pendenti
  - Application non eseguiva DB init

SOLUZIONE:
  ✅ docker-start.sh ORA:
     - Crea directory /data/molino/backups
     - Avvia direttamente app Node
     - App gestisce DB init in index.ts

CODICE CORRETTO (docker-start.sh):
  #!/bin/sh
  set -e
  echo "🚀 Starting Molino Task Manager..."
  mkdir -p /data/molino/backups
  exec node /app/server/dist/index.js

  # ❌ NON FARE: npx prisma migrate deploy
  # ❌ NON FARE: sleep 10 (aspetta!)

CODICE CORRETTO (server/src/index.ts):
  const prisma = new PrismaClient();
  
  // All startup, app gestisce DB:
  prisma.$connect().then(() => {
    console.log('✅ Database connected');
    app.listen(PORT, () => {
      console.log(`🚀 Server running on ${PORT}`);
    });
  });

PREVENZIONE:
  - docker-start.sh deve essere minimalista
  - App TypeScript gestisce Prisma operations
  - Test startup: docker-compose up -d && sleep 3 && curl http://localhost:5000/health

ERRORE #4: CSV import sovrascrive colonne importanti
---
PROBLEMA:
  - Importa CSV articoli
  - Aggiungi campo "reserved" in memoria per ordine
  - Re-import cancella "reserved" dal DB

ROOT CAUSE:
  - CSV import usa .map() per creare records, non distingue update da insert
  - Overwrite incondizionato di tutti i campi

SOLUZIONE:
  ✅ Usa prisma.upsert() con WHERE unico:
    await prisma.inventoryItem.upsert({
      where: { codice: row.codice },
      update: {
        descrizione: row.descrizione,
        categoria: row.categoria,
        // ✅ NON TOCCARE: unita, minimo (da altro processo)
      },
      create: { ...row }
    });

PREVENZIONE:
  - CSV import = update SOLO campi da CSV
  - Mantieni altri campi intatti
  - Specifica esplicitamente campi in .update({ ... })

---

ERRORE #5: CSV parser non gestisce BOM e encoding
---
PROBLEMA:
  - CSV salvato in UTF-8 con BOM (Windows)
  - Prima colonna ha caratteri strani: ïnome» invece di "nome"

ROOT CAUSE:
  - csv-parser non rimuove BOM automaticamente
  - Headers corrotti

SOLUZIONE:
  ✅ Leggi file con encoding specifico:
    import { createReadStream } from 'fs';
    fs.createReadStream(filepath, { encoding: 'utf8' })
      .pipe(csv())
      .on('data', ...)

  ✅ O pulisci prima:
    const cleaned = csvText.replace(/^\uFEFF/, '');

PREVENZIONE:
  - Salva CSV sempre in UTF-8 WITHOUT BOM
  - Testa con file reali prima di production
  - Log prima colonna header per debug

---

ERRORE #6: CSV con special characters causa 500
---
PROBLEMA:
  - Articolo con nome: "Olio 10W-40 (speciale)"
  - Import fallisce con charset error

ROOT CAUSE:
  - Virgolette non escaped nel CSV
  - Parser confuso da delimitatori

SOLUZIONE:
  ✅ CSV corretto:
    "ART001","Olio 10W-40 (speciale)","Lubrificanti",...
    (sempre tra virgolette se contiene virgole/special chars)

  ✅ Usa libreria csv con opzioni:
    csv({ delimiter: ',', quote: '"', escape: '"' })

PREVENZIONE:
  - Sempre quote CSV fields
  - Test con dati contenenti: parentesi, virgole, quote
  - Valida CSV prima di import

ERRORE #8: 401 Unauthorized su company-settings.html
---
PROBLEMA:
  - Login funziona
  - Admin dashboard carica
  - company-settings.html: tutte le richieste API ritornano 401

ROOT CAUSE:
  - API_URL hardcodato a localhost
  - Se browser accede da 127.0.0.1, API_URL = localhost → CORS fallisce
  - JWT token non inviato in header

SOLUZIONE:
  ✅ API_URL dinamico:
    const API_URL = `http://${window.location.hostname}:5000`;
    // Ora funziona da localhost, 127.0.0.1, IP, ecc
    
  ✅ JWT token in header:
    const token = localStorage.getItem('token');
    fetch(`${API_URL}/api/...`, {
      headers: {
        'Authorization': `Bearer ${token}`,
        'Content-Type': 'application/json'
      }
    })

PREVENZIONE:
  - Sempre usa window.location.hostname
  - Verifica localStorage.getItem('token') prima di API call
  - Test da 3 URL: localhost, 127.0.0.1, actual IP

ERRORE #12: Build context enorme, build molto lento
---
PROBLEMA:
  - `docker compose build` impiega 5+ minuti
  - Ogni change ricompila tutto

ROOT CAUSE:
  - .dockerignore vuoto o assente
  - Docker invia 500MB+ di context
  - Copia node_modules inutilmente

SOLUZIONE:
  ✅ .dockerignore esteso (60+ entries):
    node_modules
    dist
    .git
    README*.md
    server/prisma/migrations
    backups
    .vscode
    test
    *.db
    (... vedi file completo)

  ✅ Dockerfile strati ordinati:
    1. COPY package.json
    2. RUN npm install
    3. COPY code
    4. RUN npm run build
    
    Così: cambiamenti a HTML non ricompila npm!

RISULTATO:
  - Primo build: 1-2 minuti (completo)
  - Cambio HTML: 10-15 secondi (cache hit on npm!)

PREVENZIONE:
  - .dockerignore sempre presente
  - Ordina Dockerfile layers: stable PRIMO
  - Test build time prima/dopo

    const secret = process.env.JWT_SECRET || 'fallback-error';

  ✅ .gitignore:
    .env
    .env.local
    .env.*.local

PREVENZIONE:
  - Sempre usa .env per secrets
  - Genera JWT_SECRET: openssl rand -base64 32
  - Verifica token decode per debug (mai in logs)
